# 9. Publishing the assembly JAR

The **assembly** module produces a single JAR (`csp-sdk-<version>.jar`) that contains all connectors and their dependencies (except those excluded in the shade plugin â€” e.g., Spring Boot and `csp-sdk-core`, which are provided by the OvalEdge runtime). This JAR is placed in the External jar path of the OvalEdge application so it can discover and use all connectors.

- Build: `./build-csp-sdk.sh` or `mvn clean install -pl assembly -am`
- Output: `assembly/target/csp-sdk-<version>.jar`

## Submitting a connector for the OvalEdge Public Catalog

If you wish to contribute your connector to OvalEdge's public connector catalog (making it available to all OvalEdge customers):

1. Ensure your connector is complete, tested, and passes all local builds.
2. Raise a Pull Request against the `dev` branch of the `oe_csp_sdk` repository.
3. GitHub Actions will automatically run the connector validation workflow â€” you must resolve any failing checks before the PR can be reviewed.
4. The OvalEdge Connectors team will conduct a manual code review covering security, code quality, and connector contract compliance.
5. Upon approval and merge, OvalEdge will coordinate deployment to the public catalog and notify you of the release.

> Connector certification by OvalEdge does not transfer IP ownership for Partner-built connectors. Partners retain ownership of their connectors; OvalEdge receives a license to distribute them. See the SDK Developer Agreement for details.

# 9.1. Troubleshooting

| Issue | What to check |
|---|---|
| Connector not in `/v1/info` | SPI file present under `META-INF/services/` with the exact interface name and one FQCN per line. Ensure the connector dependency is added to `csp-api` (and `assembly`). Rebuild `csp-api` and `assembly`. |
| `csp-sdk-core` not found | Build and install `oe_csp_sdk_core` first (or publish it to the Maven repository your build uses). Ensure your Maven `settings.xml`/POM has access to the repository hosting `csp-sdk-core`. |
| JDBC connection fails | Ensure `ConnectionConfig` includes `jdbcUrl` and `username/password`. These must be set in the connector from attributes before calling `ConnectionPoolManager`. See `MonetDB`'s `ensureConnectionConfig`. |
| Assembly build fails | Run `mvn clean install` from the root first so all connector modules are installed. Then run `./build-csp-sdk.sh` or `mvn clean install -pl assembly -am`. |
| Port already in use | Change `server.port` in `csp-api/src/main/resources/application.properties` (default is `8800`). |
| Credentials not found at runtime | Ensure JFrog credentials are configured in your Maven `settings.xml` under the correct `<server>` ID matching the repository ID in the POM. |
| `@SdkConnector` annotation not recognized | Ensure `csp-sdk-core` is on the compile classpath, not just the test classpath. Check the dependency scope in your connector's `pom.xml`. |

# 12. Deployment Checklist (Partner Runtime)

Use this checklist after connector development is complete and before handing off to operations.

**1. Build a deployable artifact**

Generate `assembly/target/csp-sdk-<version>.jar` using `./build-csp-sdk.sh` or `mvn clean install -pl assembly -am`. The runtime supplies shared components (`csp-sdk-core`, Spring, `oe-core`) to prevent version clashes.

**2. Upload / copy JAR for OvalEdge pickup**

Copy the artifact to the runtime-configured jar path used by OvalEdge for connector loading. The exact folder is environment-specific and configured in OvalEdge system settings (`ovaledge.jarpath`).

**3. Ensure runtime prerequisites**

Required shared dependencies must be available in the runtime classpath (e.g., `csp-sdk-core` and platform-shared bundles). Connector JAR should include `release-{artifactId}.properties` metadata.

**4. Restart services**

Restart OvalEdge services so new connectors are discovered.

**5. Verify deployment**

- Confirm the connector appears on the Connectors page in OvalEdge.
- Run connection validation, metadata calls, and one sample query.
- Confirm release/version metadata is visible in the connector audit/release history views.

**6. Rollback**

Keep the previously known-good JAR. If validation fails after deployment, restore the old JAR and restart services. Document the rollback and open a bug report.

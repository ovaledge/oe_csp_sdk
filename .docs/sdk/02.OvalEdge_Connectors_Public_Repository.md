# 2. OvalEdge Connectors Public Repository

The OvalEdge Connectors SDK provides a prebuilt sample repository to develop new connectors.

**https://github.com/ovaledge/oe_csp_sdk**

**Fork this repository** into your own GitHub account before beginning development. Clone and push to your fork (`origin`); do not commit directly to the public repository.

> **Staying in sync with upstream:** This workflow keeps your **personal fork** aligned with the master public SDK at [https://github.com/ovaledge/oe_csp_sdk](https://github.com/ovaledge/oe_csp_sdk) as it evolves.
>
> - **`origin`** — your fork on GitHub (where you develop and push).
> - **`upstream`** — the public OvalEdge repository above (read-only source of SDK updates).
>
> Periodically pull SDK changes into your fork by adding `upstream`, fetching, and rebasing your fork’s **`main`** branch onto the public repo’s **`main`** (`upstream/main` → `origin/main`):
>
> ```bash
> git remote add upstream https://github.com/ovaledge/oe_csp_sdk.git
> git fetch upstream
> git checkout main
> git rebase upstream/main
> git push origin main   # Use --force-with-lease if you already pushed main before rebasing
> ```
>
> Resolve any conflicts before continuing development. If you work on feature branches, update them from your refreshed fork `main` after syncing (for example, `git checkout my-feature && git rebase main`). Breaking changes are documented in the SDK `CHANGELOG.md`.

# 10. Definition of Done and Success Criteria

Use this page as the single checklist to confirm a connector is complete. It consolidates implementation quality (**Definition of Done**), local verification, PR submission, and private runtime deployment.

For interface contracts and method behavior, see the [Connector Interface Reference](../Connector_Interface_Reference.md). For build, test, and review process details, see [Development & Testing](07.Development_And_Testing.md).

---

## 10.1. Definition of Done

A connector meets the **Definition of Done** when all of the following are true:

- [ ] Implements all required methods on `AppsConnector`, `MetadataService`, and `QueryService` per the [Connector Interface Reference](../Connector_Interface_Reference.md).
- [ ] SPI registration is present at `META-INF/services/com.ovaledge.csp.v3.core.apps.service.AppsConnector` with the correct FQCN.
- [ ] `getServerType()` is unique, lowercase, and stable.
- [ ] Object identity is consistent across metadata and query flows (same IDs/keys in `getObjects()` and `fetchData()`).
- [ ] Connection attribute keys and `exchangeAttributes()` mapping are backward-compatible.
- [ ] Edge cases are handled; client-visible error messages are clear and sanitized (see [§5.4 Error Handling](05.What_you_will_develop.md#54-error-handling-conventions)).
- [ ] Behavior is thread-safe under concurrent requests.
- [ ] Logging follows [§5.5 Logging Guidelines](05.What_you_will_develop.md#55-logging-guidelines); no credentials, tokens, or PII in logs.
- [ ] Module `README.md` documents developer setup, supported object types, and connector-specific limits or unsupported features.
- [ ] End-user connector documentation is complete using the [OvalEdge Connector Documentation Template](../templates/OvalEdge_Connector_Documentation_Template.md) (see [Connector documentation](../OvalEdge_Connectors_Software_Development_Kit.md#connector-documentation) in the SDK guide).
- [ ] `@SdkConnector` annotation and release metadata file are included where applicable.

---

## 10.2. Success Criteria — Local Verification

Before assembling, publishing, or raising a PR, confirm:

- [ ] `mvn clean install` passes locally with no errors.
- [ ] The connector appears in `GET /v1/info` when running `csp-api` locally.
- [ ] `validateConnection()` succeeds with valid test credentials.
- [ ] `getSupportedObjects()`, `getContainers()`, `getObjects()`, and `getFields()` return expected results.
- [ ] `fetchData()` returns expected results for at least one supported entity type.
- [ ] No credentials, tokens, or API keys are hardcoded in source or test fixtures committed to the repo.

---

## 10.3. Success Criteria — Public Catalog (Pull Request)

Required only when submitting a connector to the OvalEdge public catalog. Full gate details are in [§7.3 Code Review Standards](07.Development_And_Testing.md#73-code-review-standards-and-quality-gates).

- [ ] Module is wired into the parent POM, `csp-api`, and `assembly` (see [Assemble the new connector](08.Assemble_the_new_connector.md)).
- [ ] All GitHub Actions checks pass (build, tests, SAST, dependency scan, contract validation, SPI check, secrets scan).
- [ ] Unit tests cover happy path, errors, and edge cases for public connector methods.
- [ ] PR description includes: connector name, target data source, test evidence, and known limitations.
- [ ] Connector documentation (template-based) is included in the delivery and linked from the module `README.md`.

---

## 10.4. Success Criteria — Private Deployment

Required when deploying a **private** connector to an OvalEdge runtime. Operational steps are in [§12 Deployment Checklist](09.Publishing_the_assembly_JAR.md#12-deployment-checklist-partner-runtime).

- [ ] `assembly/target/csp-sdk-<version>.jar` is built and copied to the runtime jar path (`ovaledge.jarpath`).
- [ ] OvalEdge services are restarted after deployment.
- [ ] Connector appears on the Connectors page; connection validation, metadata, and sample query succeed in the target environment.
- [ ] A rollback plan exists (previous known-good JAR retained).

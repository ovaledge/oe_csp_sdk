# 5. What you will develop

A connector is a **Maven module** that:

1. Implements **AppsConnector** (connection validation, metadata, query, server type, connection attributes).
2. Provides **MetadataService** (supported object types, containers, objects, fields).
3. Provides **QueryService** (executes queries and fetches data for an entity).
4. Is registered in `META-INF/services/com.ovaledge.csp.v3.core.apps.service.AppsConnector` so the SDK and `csp-api` can discover it via ServiceLoader.

Optionally, it can include a **quick** package: a small Spring Boot app to run and test that connector alone (e.g., health, connection validate, supported-objects). Review the `monetdb` and `awsconsole` connectors as reference implementations.

## 5.1 Connector Contract

For implementation, treat the [Connector Interfaces Reference](../Connector_Interface_Reference.md) as the source of truth for:

- Required vs. optional/default methods
- Method inputs, outputs, errors, and invariants
- Lifecycle execution order
- Thread-safety and performance expectations

## 5.2 Connector Interfaces

### 5.2.1 AppsConnector (from csp-sdk-core)

Your connector class must implement:


| Method                                                     | Purpose                                                                                |
| ---------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| `getServerType()`                                          | Unique string used in requests (e.g., `"monetdb"`, `"myapp"`). Use lowercase.          |
| `validateConnection(ConnectionConfig config)`              | Validate credentials/settings; return success/failure.                                 |
| `getMetadataService()`                                     | Return your `MetadataService` implementation.                                          |
| `getQueryService()`                                        | Return your `QueryService` implementation.                                             |
| `getAttributes()`                                          | Connection form definition (host, port, username, password, etc.) for the OvalEdge UI. |
| `exchangeAttributes(ConnInfo)` / `exchangeAttributes(Map)` | Map between UI attributes and ConnInfo / config.                                       |


Additional methods (`getExtendedAttributes`, `exchangeVaultAttributes`, `getVaultPath`, etc.) are used for secrets/vault and governance; you can delegate to `BaseAppConnector` for common behavior (see `monetdb`).

### 5.2.2 MetadataService


| Method                             | Purpose                                                                   |
| ---------------------------------- | ------------------------------------------------------------------------- |
| `getSupportedObjects()`            | List supported entity types (e.g., entities, reports).                    |
| `getContainers(ContainersRequest)` | List top-level containers (e.g., schemas, companies, accounts).           |
| `getObjects(ObjectRequest)`        | List objects (e.g., tables, entities, reports) in a container for a type. |
| `getFields(FieldsRequest)`         | List fields (columns) for a given object.                                 |


### 5.2.3 QueryService


| Method                    | Purpose                                                             |
| ------------------------- | ------------------------------------------------------------------- |
| `fetchData(QueryRequest)` | Execute the query (filters, fields, limit, offset) and return rows. |


## 5.3 Required vs. Optional (Implementation Checklist)

- **Required to implement:** `AppsConnector`, `MetadataService`, `QueryService` contracts used by runtime routing and query flow.
- **Optional/default:** EDGI and SDK version methods when the connector does not support those workflows.
- **Strongly recommended:** Annotate the connector with `@SdkConnector(artifactId="...")` and include a release metadata file for runtime auditability.

## 5.4 Error Handling Conventions

Connectors must handle errors predictably so the OvalEdge runtime can report meaningful messages to end users. Follow these conventions:

- **Connection failures:** Throw or return a structured error response from `validateConnection()` with a human-readable message. Do not swallow exceptions silently.
- **Metadata errors:** If a container or object cannot be listed (e.g., insufficient permissions), return an empty result with a log warning rather than throwing. Only throw for fatal, unrecoverable errors.
- **Query errors:** Return structured error responses from `fetchData()` including the error message and, where possible, the failing query or entity reference.
- **Timeouts:** Honor the timeout values in `ConnectionConfig` or `QueryRequest`. Surface timeout exceptions with a clear message, including elapsed time.
- **Unchecked exceptions:** Wrap all third-party SDK or JDBC exceptions in OvalEdge connector exception types. Do not propagate driver-specific exceptions to the runtime.

## 5.5 Logging Guidelines

Use the SLF4J logger provided by the SDK base classes. Follow these conventions:


| Level   | Use for                                                                                   |
| ------- | ----------------------------------------------------------------------------------------- |
| `ERROR` | Unrecoverable failures (authentication failure, network unreachable, fatal config error). |
| `WARN`  | Recoverable issues (empty result set, deprecated config field, retried connection).       |
| `INFO`  | Connection lifecycle events (connected, disconnected, query completed).                   |
| `DEBUG` | Request/response details, query parameters, field mappings â€” enabled only in development. |


**Never log credentials, tokens, passwords, or personally identifiable information (PII) at any log level.** Mask sensitive values before logging connection attributes.

## 5.6 SDK scope and limitations

SDK connectors are built for **connection validation**, **metadata discovery** (catalog/crawl), and **sample data preview** in the Data Catalog. Other OvalEdge capabilities exist on the platform but are **not available for SDK connectors in the current release**.

### 5.6.2 What you must define for your source

The SDK does not automatically support every feature your source offers. Document these in your connector README:

| Topic | Your responsibility |
| ----- | ------------------- |
| **Object types** | Only object types you implement are cataloged (for example, tables only, or tables plus reports). |
| **Incremental extraction** | If the source has no delta API, state that OvalEdge runs **full crawl** on a schedule. |
| **Authentication** | List which sign-in methods work (and which do not) for your connection form. |
| **Data reads** | Sample Data uses structured reads for a **selected catalog object**, not free-form SQL or REST from end users. |
| **Crawler options** | Optional crawl features (reports, files, APIs, etc.) must be implemented **and** registeredâ€”users only see what you enable. |

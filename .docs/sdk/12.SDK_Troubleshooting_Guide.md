# 12. SDK Troubleshooting Guide (Connector Consumer Problems)

This guide documents the problems users may hit while consuming the SDK and building OvalEdge connectors, written from the perspective of a Partner/Customer connector developer: registration → fork → build/test → deliver an assembly JAR for OvalEdge integration.

The goal is to help you quickly map a symptom to the most likely cause, then apply the smallest safe fix and validate in `csp-api`.

---

## Registration/Access

### JFrog credentials not available (Maven dependency downloads fail)
**Severity:** Critical
**Symptoms:**
- `mvn clean install` fails with `401`, `403`, or “artifact not found” for SDK components
- Builds succeed only on machines where SDK credentials were already configured
**Possible Causes:**
- You started building before SDK developer registration completed
- Your Maven `settings.xml` has no `<server>` entry for the JFrog repo(s)
**Troubleshooting checklist:**
- Confirm registration is complete (credentials issued) for your developer persona
- Verify `settings.xml` is being used by Maven (`mvn help:effective-settings | rg server`)
- Confirm the `<server>` `id` matches the repository id referenced in the connector POM(s)
**Fix steps:**
- Wait for credentials provisioning if registration is still pending
- Add/repair `<server>` credentials in `settings.xml` and re-run the build
**Validation:**
- `mvn clean install` progresses past dependency resolution
**References:**
- [SDK prerequisites](01.Prerequisites.md): “OvalEdge SDK Developer Registration … JFrog credentials are issued after registration is complete”

### SDK build fails immediately due to wrong Java version (not JDK 21)
**Severity:** High
**Symptoms:**
- `mvn` fails with compiler/toolchain errors (e.g., unsupported release)
- Builds fail in CI but not on one developer machine (or vice versa)
**Possible Causes:**
- `JAVA_HOME` points to a JDK other than 21
- Your shell/session uses a different JDK than your IDE
**Troubleshooting checklist:**
- Check `java -version` and `mvn -v`
- Ensure Maven compiler plugin is aligned with the intended JDK
**Fix steps:**
- Set `JAVA_HOME` to JDK 21
- Restart the terminal/IDE to pick up the correct environment
**Validation:**
- Maven compile succeeds locally with the same commands CI uses
**References:**
- [Prerequisites](01.Prerequisites.md): “Java | JDK 21 — ensure `JAVA_HOME` points to JDK 21”

### SDK version mismatch vs OvalEdge platform (connector loads inconsistently)
**Severity:** High
**Symptoms:**
- Connector builds but fails contract validation or fails to register correctly in OvalEdge runtime
- `/v1/info` may work locally but production fails during discovery or operations
**Possible Causes:**
- You built against an SDK version that is not compatible with your target OvalEdge platform release train
- You did not verify compatibility before starting work
**Troubleshooting checklist:**
- Identify the SDK version you built (from `pom.xml` / `csp-sdk-<version>.jar`)
- Identify the target OvalEdge platform version
- Check the compatibility matrix URL before further debugging
**Fix steps:**
- Rebuild using the SDK version compatible with your target platform release train
**Validation:**
- Connector discovery and one end-to-end call (validate → metadata → query/sample data) succeeds in the target environment
**References:**
- [Prerequisites](01.Prerequisites.md): “Version Compatibility … check this before starting development”
- [Development & Testing](07.Development_And_Testing.md): “Version Compatibility … A version compatibility matrix will be published…”

### Cannot fork/sync or open PR due to Git access/permissions
**Severity:** Medium
**Symptoms:**
- `git push` fails with “permission denied” or “repository not found”
- Sync/rebase operations fail due to authentication
**Possible Causes:**
- Your GitHub credentials/token are not set up for the repo host you’re using
- You’re pushing to the wrong remote/branch
**Troubleshooting checklist:**
- Confirm remotes: `git remote -v`
- Confirm you can access the upstream repository from the browser
- Confirm the PR target branch matches the documented workflow
**Fix steps:**
- Re-authenticate to GitHub (credential manager / token)
- Ensure your PR targets the correct branch (`dev` for public catalog submission)
**Validation:**
- PR opens successfully and GitHub Actions pipeline starts running
**References:**
- [Publishing the assembly JAR](09.Publishing_the_assembly_JAR.md): “Raise a Pull Request against the `dev` branch”

### Missing required registration/onboarding artifacts delays local development
**Severity:** Medium
**Symptoms:**
- You cannot build or run the SDK template because required access artifacts are missing
- You are blocked on “setup” rather than connector code
**Possible Causes:**
- Developer onboarding steps not fully completed (agreement, provisioning, etc.)
**Troubleshooting checklist:**
- Confirm registration outcomes (agreement signed, credentials issued)
- Confirm you have the required access paths to required artifacts (JFrog repos, SDK components)
**Fix steps:**
- Complete the remaining onboarding items before debugging connector behavior
**Validation:**
- You can reach the stage where `csp-api` runs and lists your connector
**References:**
- [Prerequisites](01.Prerequisites.md): registration must be complete before proceeding

## Local Setup

### Maven build fails because Maven is missing or too old
**Severity:** High
**Symptoms:**
- `mvn: command not found`
- Maven refuses to run or plugins fail with version-related errors
**Possible Causes:**
- Maven not installed
- Maven version < 3.8+
**Troubleshooting checklist:**
- Run `mvn -v`
- Confirm the Maven version meets the documented requirement
**Fix steps:**
- Install/upgrade Maven to 3.8+
- Re-open terminal/IDE after install
**Validation:**
- `mvn clean install` runs past the initial project build
**References:**
- [Prerequisites](01.Prerequisites.md): “Maven 3.8+ — ensure `mvn -v` works”

### Build fails because `oe_csp_sdk_core` (or shared components) are not available
**Severity:** Critical
**Symptoms:**
- Errors like `csp-sdk-core` not found in Maven resolution
- `@SdkConnector` related types/classes cannot be found during compile
**Possible Causes:**
- You did not build/install `oe_csp_sdk_core` before building dependent modules
- Your Maven settings/POM cannot resolve `csp-sdk-core` from the repository it’s published to
**Troubleshooting checklist:**
- Verify `oe_csp_sdk_core` was built/installed into your local Maven repo (or reachable from remote)
- Check `mvn -pl ... -am` paths for the reactor build
**Fix steps:**
- Build and install `oe_csp_sdk_core` first, or ensure repository access in `settings.xml`
**Validation:**
- Compilation succeeds and your connector module builds successfully
**References:**
- [Publishing the assembly JAR — troubleshooting](09.Publishing_the_assembly_JAR.md): “`csp-sdk-core` not found … Build and install `oe_csp_sdk_core` first”

### `csp-api` won’t start (port already in use)
**Severity:** High
**Symptoms:**
- When running `csp-api` you get “address already in use”
- The `csp-api` UI is not reachable at `http://localhost:8800/`
**Possible Causes:**
- Another process is using port `8800`
**Troubleshooting checklist:**
- Confirm the port configured in `csp-api/src/main/resources/application.properties`
- Check which process is using the port
**Fix steps:**
- Change `server.port` in `application.properties` to a free port, then rebuild/run
**Validation:**
- `curl -s http://localhost:<port>/v1/info` returns a response
**References:**
- [Development & Testing](07.Development_And_Testing.md): default port is **8800**
- [Publishing the assembly JAR — troubleshooting](09.Publishing_the_assembly_JAR.md): “Port already in use … default is `8800`”

### Upstream sync/rebase fails with conflicts or dirty working tree
**Severity:** Medium
**Symptoms:**
- `git rebase upstream/main` fails
- You see merge conflict markers or a “cannot rebase: unstaged changes” error
**Possible Causes:**
- You have uncommitted changes before syncing
- Code has diverged from upstream
**Troubleshooting checklist:**
- Ensure working tree is clean or stash changes before sync
- Re-run the documented upstream sync procedure
**Fix steps:**
- Follow the “Syncing Changes” steps: fetch upstream, checkout `main`, rebase, push
**Validation:**
- Repository history matches upstream expectations and your build continues to work
**References:**
- [Development & Testing](07.Development_And_Testing.md): upstream sync steps + guidance

### `GET /v1/info` is unreachable (wrong URL/port or csp-api not running)
**Severity:** Medium
**Symptoms:**
- `curl http://localhost:8800/v1/info` fails (connection refused / timeout)
**Possible Causes:**
- `csp-api` not running
- Port differs from default due to local configuration
**Troubleshooting checklist:**
- Confirm `csp-api` process is running
- Confirm the port in `application.properties`
**Fix steps:**
- Start `csp-api` and use the correct port in your test commands
**Validation:**
- `GET /v1/info` responds (200) and includes your connector in discovery metadata
**References:**
- [Development & Testing](07.Development_And_Testing.md): `GET /v1/info does not require a request body`

## Generate scaffold

### Connector does not appear in `/v1/info` (SPI missing from assembly/discovery)
**Severity:** Critical
**Symptoms:**
- Your connector is absent from the discovery results when running `csp-api`
- Local UI cannot show your connector in connector selection
**Possible Causes:**
- SPI registration is missing from `META-INF/services/`
- The connector dependency is not wired into `csp-api` and/or `assembly`
**Troubleshooting checklist:**
- Confirm the SPI file exists under `META-INF/services/` for `AppsConnector`
- Confirm it has the exact interface name and correct one-FQCN-per-line entries
- Confirm you built both `csp-api` and `assembly` after wiring
**Fix steps:**
- Add/repair the SPI file and ensure connector module is added to parent POM/csp-api/assembly
- Rebuild `csp-api` and `assembly`
**Validation:**
- `curl -s http://localhost:<port>/v1/info` includes your connector `serverType`
**References:**
- [Publishing the assembly JAR — troubleshooting](09.Publishing_the_assembly_JAR.md):
  > “Connector not in `/v1/info` … SPI file present under `META-INF/services/` … Ensure the connector dependency is added to `csp-api` (and `assembly`).”

### SPI file points to the wrong interface name or wrong FQCN
**Severity:** High
**Symptoms:**
- Connector still doesn’t load even after adding SPI
- ServiceLoader errors or silent discovery failure
**Possible Causes:**
- Interface name in the SPI file is incorrect
- FQCN listed in the SPI file doesn’t match the actual connector implementation class
**Troubleshooting checklist:**
- Verify SPI file location and naming
- Verify the listed FQCN matches the implementation class name + package
**Fix steps:**
- Correct the SPI interface filename and the FQCN line(s)
- Rebuild and re-test discovery via `/v1/info`
**Validation:**
- Connector appears in `/v1/info` and a validate/metadata call returns expected responses
**References:**
- [Definition of Done](10.Definition_of_Done_and_Success_Criteria.md): “SPI registration is present … with the correct FQCN.”

### `getServerType()` returns blank/null or changes across releases
**Severity:** Critical
**Symptoms:**
- Requests routed by `serverType` fail
- Connector lookup/routing fails after a release
**Possible Causes:**
- `getServerType()` returns `null`, blank, or an unstable value
- You changed `serverType` between connector versions unintentionally
**Troubleshooting checklist:**
- Inspect the connector’s `getServerType()` implementation
- Confirm it returns a stable, non-empty lowercase string
**Fix steps:**
- Set `getServerType()` to a stable unique value and never change it for the same connector family
**Validation:**
- Discovery and subsequent operations route correctly by `serverType`
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Invariant: Value must remain constant across releases … Returning `null` or a blank value will cause registry and routing failures.”

### `exchangeAttributes(...)` mapping breaks connection configuration (validateConnection gets missing data)
**Severity:** High
**Symptoms:**
- Connection validation fails with “missing required fields” even though the UI shows values
- `validateConnection()` behavior differs between local tests and runtime
**Possible Causes:**
- Attribute keys changed or were normalized/destructively changed during exchange
- `exchangeAttributes(ConnInfo)` and `exchangeAttributes(Map)` are not inverse mappings
**Troubleshooting checklist:**
- Compare UI attributes you set with the `ConnInfo` produced by `exchangeAttributes(Map)`
- Confirm mapping preserves existing values without destructive normalization
**Fix steps:**
- Repair mapping to ensure both conversion directions are consistent
- Keep attribute keys backward-compatible
**Validation:**
- `validateConnection()` succeeds with the expected config payload in both local and runtime tests
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “exchangeAttributes(ConnInfo) … must preserve existing values without destructive normalization.”

### `@SdkConnector` annotation not recognized / release metadata missing
**Severity:** Medium
**Symptoms:**
- Release metadata doesn’t show correctly
- Contract checks around annotation presence fail
**Possible Causes:**
- `csp-sdk-core` is missing from compile classpath (annotation only available in tests)
- Release metadata file not included
**Troubleshooting checklist:**
- Verify dependency scope for `csp-sdk-core`
- Verify `release-csp-sdk.properties` is packaged into the final jar
**Fix steps:**
- Ensure annotation dependency is on compile classpath
- Include release metadata file
**Validation:**
- Runtime audit/release history includes expected metadata
**References:**
- [Publishing the assembly JAR — troubleshooting](09.Publishing_the_assembly_JAR.md):
  > “`@SdkConnector` annotation not recognized … Ensure `csp-sdk-core` is on the compile classpath…”
- [Definition of Done](10.Definition_of_Done_and_Success_Criteria.md): “`@SdkConnector` annotation and release metadata file are included where applicable.”

## Develop & test

### `validateConnection()` fails (success=false) in a way that blocks connector usage
**Severity:** Critical
**Symptoms:**
- Users cannot validate and save the connection in OvalEdge UI
- Error message is unclear, or it leaks internal details
**Possible Causes:**
- Missing required connection fields for your auth flow
- Connectivity/auth problems not bounded or misreported
**Troubleshooting checklist:**
- Confirm required attributes are validated and errors are returned as structured failure responses
- Confirm you’re not throwing unrecoverable exceptions for expected validation problems
**Fix steps:**
- Ensure `validateConnection()` returns structured failure (`success=false`) with a safe message/errorCode
- Add clear input validation and bounded checks (no infinite waits)
**Validation:**
- Local validation passes with correct credentials; runtime validation passes with the target environment config
**References:**
- [What you will develop — error handling](05.What_you_will_develop.md):
  > “Connection failures: Throw or return a structured error response from `validateConnection()` with a human-readable message.”

### Metadata discovery returns empty containers/objects unexpectedly
**Severity:** High
**Symptoms:**
- Catalog crawl shows no schemas/tables/fields for the connector
- Calls like `getContainers()` / `getObjects()` return empty lists without obvious runtime errors
**Possible Causes:**
- `entityType` values returned by `getSupportedObjects()` don’t match what `getContainers()/getObjects()` expect
- Inconsistent object identity semantics across methods (IDs/keys don’t line up)
- Permissions/auth issues at metadata time
**Troubleshooting checklist:**
- Verify `getSupportedObjects()` returns the exact `typeName` values you later accept in `getContainers()` and `getObjects()`
- Confirm `getContainers/getObjects/getFields` are driven by the same deterministic identity semantics
- For permissions issues, check logs for the “return empty result with log warning” behavior
**Fix steps:**
- Align `entityType` handling across supported objects → containers → objects → fields
- Ensure object identity keys are consistent between metadata and query flows
**Validation:**
- `getSupportedObjects()`, `getContainers()`, `getObjects()`, and `getFields()` return expected non-empty results for a known test connection
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Ordering Invariants … `getContainers()` and `getObjects()` must consistently accept and handle the same `entityType` values …”
- [What you will develop — metadata errors](05.What_you_will_develop.md):
  > “Metadata errors … return an empty result with a log warning rather than throwing.”

### `fetchData()` fails or returns rows that don’t match `getFields()`/field projections
**Severity:** High
**Symptoms:**
- Queries fail with schema-related errors
- Returned data has missing/extra keys compared to expected field names/types
- Pagination looks wrong (unexpected totalRows/row counts)
**Possible Causes:**
- Row map keys are inconsistent with field keys produced by `getFields()`
- `columnNames` order doesn’t match the row maps
- `limit/offset` semantics are inconsistent with documentation
**Troubleshooting checklist:**
- Verify the requested fields are reflected in your row mapping
- Confirm row maps use stable field keys consistent with `getFields()`
- Validate pagination behavior against a small deterministic dataset
**Fix steps:**
- Repair mapping from domain data → `List<Map<String,Object>>` keys to match `getFields()` field names
- Ensure `columnNames` order reflects the produced row schema
**Validation:**
- A sample query returns `data` with all expected keys and correct `totalRows` for the dataset size
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Row maps use stable field keys consistent with `getFields` … Pagination semantics (limit/offset) are consistent…”
- [What you will develop — query errors](05.What_you_will_develop.md):
  > “Query errors: Return structured error responses from `fetchData()` including the error message…”

### CI/test fails due to secrets/credentials in logs or code (sanitization failure)
**Severity:** Critical
**Symptoms:**
- GitHub Actions fails with “hardcoded secrets” or security scan failures
- Manual review flags credentials/token/PII logging
**Possible Causes:**
- You log sensitive material in exceptions or logs
- You include secret payloads in error messages
**Troubleshooting checklist:**
- Search for likely sensitive patterns (`password`, `token`, `secret`, `authorization`)
- Check exception messages and log statements for credential leakage
**Fix steps:**
- Remove or mask sensitive values before logging/returning messages
- Ensure exceptions/messages are sanitized and stable
**Validation:**
- Security scan passes; manual review is satisfied on logging hygiene
**References:**
- [What you will develop — logging](05.What_you_will_develop.md):
  > “Never log credentials, tokens, passwords, or personally identifiable information (PII) at any log level.”

## Assemble

### Assembly build fails (reactor/module order or missing installed modules)
**Severity:** Critical
**Symptoms:**
- `./build-csp-sdk.sh` fails or produces no `csp-sdk-<version>.jar`
- `mvn clean install -pl assembly -am` fails due to missing module builds
**Possible Causes:**
- You didn’t build/install upstream modules first
- Partial builds skipped required dependencies
**Troubleshooting checklist:**
- Use the documented build command from repo root
- Confirm that dependent modules are built as part of the reactor
**Fix steps:**
- Run from repo root: `mvn clean install` (or `mvn clean install -pl assembly -am`) then re-run assembly build
**Validation:**
- `assembly/target/csp-sdk-<version>.jar` exists and is non-empty
**References:**
- [Publishing the assembly JAR](09.Publishing_the_assembly_JAR.md):
  > “Assembly build fails … Run `mvn clean install` from the root first … then run `./build-csp-sdk.sh` …”

### Expected output jar is missing or copied from the wrong path
**Severity:** High
**Symptoms:**
- OvalEdge runtime can’t load your connector jar
- Deploy step reports jar not found
**Possible Causes:**
- You copied from the wrong module folder
- Build produced a jar with a different version string
**Troubleshooting checklist:**
- Confirm the actual jar output path: `assembly/target/csp-sdk-<version>.jar`
- Confirm you’re using the jar you just rebuilt (not an older one)
**Fix steps:**
- Copy the correct `csp-sdk-<version>.jar` to the runtime-configured jar path
**Validation:**
- Runtime discovery picks up the connector after service restart
**References:**
- [Publishing the assembly JAR](09.Publishing_the_assembly_JAR.md):
  > “Output: `assembly/target/csp-sdk-<version>.jar`”

### `release-csp-sdk.properties` missing from the final assembly jar
**Severity:** Medium
**Symptoms:**
- Release metadata isn’t visible in runtime audit/release history views
- Internal checks complain about missing release properties
**Possible Causes:**
- Release metadata file not included in connector module/jar packaging
**Troubleshooting checklist:**
- Verify packaging includes `release-csp-sdk.properties`
**Fix steps:**
- Add/repair the release metadata file inclusion in the relevant module(s)
- Rebuild assembly
**Validation:**
- Runtime UI shows release/version metadata for the connector
**References:**
- [Definition of Done](10.Definition_of_Done_and_Success_Criteria.md):
  > “Connector JAR should include `release-csp-sdk.properties` metadata.”

### Runtime classes missing due to incorrect shading/dependency wiring
**Severity:** High
**Symptoms:**
- Runtime fails with `ClassNotFoundException` or similar linkage errors
- Errors show up only after deploying the assembly jar
**Possible Causes:**
- Required dependencies are excluded unintentionally by the shade plugin/config
- Connector module dependencies aren’t wired into assembly correctly
**Troubleshooting checklist:**
- Confirm your connector dependencies are included as intended by the assembly build
- Re-check that excluded libraries (like `csp-sdk-core`) are provided by runtime
**Fix steps:**
- Adjust shade/dependency configuration so required runtime types are present
**Validation:**
- Deploy jar, restart, then validate `/v1/info` and a basic validate/metadata call
**References:**
- [Publishing the assembly JAR](09.Publishing_the_assembly_JAR.md):
  > “assembly … except those excluded in the shade plugin … provided by the OvalEdge runtime”

### build-csp-sdk script fails due to environment or permissions
**Severity:** Medium
**Symptoms:**
- `./build-csp-sdk.sh: permission denied`
- Script execution fails due to missing environment variables
**Possible Causes:**
- Script not executable
- Environment/setup differs from expectations
**Troubleshooting checklist:**
- Confirm script exists and is executable
- Compare environment used by CI/local
**Fix steps:**
- Ensure script executable permissions and required environment variables
- Alternatively run the equivalent Maven command
**Validation:**
- assembly artifact is produced at the expected path
**References:**
- [Publishing the assembly JAR](09.Publishing_the_assembly_JAR.md):
  > “Build: `./build-csp-sdk.sh` or `mvn clean install -pl assembly -am`”

### Credentials/secrets missing (vault path/secrets not present) causes validateConnection failures
**Severity:** High
**Symptoms:**
- Connections fail during validation in runtime (but not locally)
- Errors indicate missing secret/vault entries or inability to resolve secret keys
**Possible Causes:**
- Selected credential manager configuration doesn’t include the required secret/vault values
- Vault path or secret name mismatch between connector expectations and runtime configuration
**Troubleshooting checklist:**
- Confirm the connector’s secret/vault fields are mapped correctly in `exchangeVaultAttributes` and related flows
- Confirm the runtime credential manager has the expected secrets at the expected path
**Fix steps:**
- Update the connector’s secret key expectations to match the intended runtime credential manager schema
- Ensure required secrets are present in the target secret store and credential manager configuration is correct
**Validation:**
- Connection validate succeeds in runtime and metadata/query calls proceed
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Security: avoid leaking sensitive data (credentials/secrets) in exceptions, messages, or logs.”
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Credential handling … optionally managed by credential-manager/vault helpers within `BaseAppConnector`.”

### Metadata/query identity mismatch causes “empty metadata” or “field missing” failures
**Severity:** High
**Symptoms:**
- `getFields()` returns fields, but `fetchData()` returns rows that don’t match requested fields
- Or: metadata seems incomplete, then query fails for entity/container/entity references
**Possible Causes:**
- Object identity semantics are inconsistent between `getFields()` and `fetchData()`
- You treat `entityType/containerId/entityId` differently across metadata and query flows
**Troubleshooting checklist:**
- Confirm the same identity keys are used across metadata and query flows
- Validate that `entityType` values are stable and consistent with `getSupportedObjects()`
**Fix steps:**
- Repair object identity mapping and ensure invariants hold for all three of `entityType`, `containerId`, `entityId`
**Validation:**
- A query for a previously crawled object returns data rows with matching keys
**References:**
- [Connector Interface Reference](../Connector_Interface_Reference.md):
  > “Ordering Invariants … object identity semantics … used by `getFields()` must be identical to those used by `fetchData()`.”

### Runtime connectivity/timeouts (network/whitelisting/firewall) break validate/metadata/query
**Severity:** High
**Symptoms:**
- Runtime errors mention timeouts, unreachable hosts, or auth failures tied to connectivity
- Validate works locally but fails in the target environment
**Possible Causes:**
- Required ports are not reachable from OvalEdge runtime
- Service account/vault credentials point to wrong environment endpoints
**Troubleshooting checklist:**
- Verify whitelisting/egress rules allow outbound/inbound traffic for the connector
- Verify `ConnectionConfig` fields are correct for the runtime environment
- Check timeout handling and message clarity
**Fix steps:**
- Update network/port whitelisting for OvalEdge runtime
- Correct environment-specific endpoints/configuration values
- Ensure you surface timeout errors with a clear, sanitized message
**Validation:**
- Connection validate succeeds and subsequent metadata/query calls return expected results
